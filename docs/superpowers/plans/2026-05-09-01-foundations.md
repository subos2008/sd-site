# Plan 01 — Project Foundations

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the empty SD Site project with all infrastructure needed for subsequent plans — frontend bootstrap, backend bootstrap, shared config/contract chain, test harnesses across all layers, observability, i18n, PWA plumbing, and a green CI pipeline.

**Architecture:** Vite + React + TypeScript SPA in `src/`. Supabase backend (managed in `supabase/`, run locally via Supabase CLI in Docker). Cross-cutting source files (config, RPC contracts, generated DB types) live in `/shared/` and are imported by both `src/` and `supabase/functions/`. All automated tests run against ephemeral Docker Supabase. CI exercises the full chain on every PR.

**Tech Stack:** Vite, React 18, TypeScript, Tailwind CSS, react-router, react-i18next, vite-plugin-pwa, Zod, OpenTelemetry (web + Deno SDKs, no-op when env unset), Vitest, React Testing Library, MSW, Playwright, pgTAP, Supabase CLI (Docker), pnpm, GitHub Actions.

**Outcome at the end of this plan:** the project builds, all five test layers run green (Vitest unit, Vitest component, MSW-mocked, pgTAP, Playwright), local Supabase starts and applies migrations, the shared config file flows correctly into both the frontend bundle and the Postgres `app_config` table, Zod contracts are wired, OTEL initialises (no-op), i18n returns English strings, react-router renders placeholder routes, the PWA manifest is served, and CI runs all of the above.

**No user-facing features in this plan.** That's Plan 02.

### Versions actually installed during execution

This plan was originally written assuming older tooling. Execution adapted to current versions. Notable differences:

- **React 19** + **TypeScript 6** + **Vite 8** (not React 18 / TS 5 / Vite 5)
- **Tailwind CSS v4** with `@tailwindcss/vite` plugin (not v3 with PostCSS) — no `tailwind.config.ts` or `postcss.config.cjs` needed
- **ESLint 9 flat config** (`eslint.config.js`) — not legacy `.eslintrc.cjs`
- **pnpm 11** with explicit `pnpm-workspace.yaml` `allowBuilds` map for packages with postinstall scripts (esbuild, msw, protobufjs)
- **Modern tsconfig split**: `tsconfig.json` is references-only; app code config lives in `tsconfig.app.json`; path aliases go there
- **TypeScript 6 deprecates `baseUrl`** — use explicit relative paths in the `paths` map (e.g., `"./src/*"`)
- **OpenTelemetry new SDK**: `resourceFromAttributes(...)` (not `new Resource(...)`), `spanProcessors:` constructor option (not `provider.addSpanProcessor`), `ATTR_SERVICE_NAME` constants (not `SemanticResourceAttributes`)
- **React Router v7** installed; v6 `createBrowserRouter` syntax still works as-is
- **Supabase CLI 2.78.1**: in `supabase/config.toml` set `[analytics] enabled = false` to avoid local Logflare healthcheck failures
- **MSW v2** (uses `http.post(...)`)
- **Node 22**, not Node 20 — update CI matrix accordingly

## Task 3 — Path aliases (`@/`, `@shared/`)

**Files:**

- Modify: `tsconfig.json`, `vite.config.ts`
- Create: `shared/.gitkeep` (placeholder; real files come later)

- [ ] **Step 1: Add path aliases to `tsconfig.app.json` (NOT tsconfig.json)**

The modern Vite template splits TypeScript config: `tsconfig.json` is references-only, and `tsconfig.app.json` is where `src/` code's compiler options live. Path aliases must go in `tsconfig.app.json`.

In its `compilerOptions`, add:

```json
{
  "paths": {
    "@/*": ["./src/*"],
    "@shared/*": ["./shared/*"]
  }
}
```

Note: do NOT add `baseUrl`. TypeScript 6.x deprecates it and `tsc -b` will fail with TS5101. Modern TypeScript resolves `paths` relative to the tsconfig location when `baseUrl` is absent — use explicit `./src/*` and `./shared/*` paths.

- [ ] **Step 2: Add matching aliases to `vite.config.ts`**

```ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import path from 'node:path'

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
      '@shared': path.resolve(__dirname, './shared'),
    },
  },
})
```

## Task 9 — Build script: `gen:config` produces SQL seed migration from shared config

**Files:**

- Create: `scripts/gen-config.mjs`
- Modify: `package.json` (scripts)
- Generated (re-runnable): `supabase/migrations/20260509000001_app_config_seed.sql`

- [ ] **Step 1: Write the generator script**

Create `scripts/gen-config.mjs`:

```js
#!/usr/bin/env node
// Reads shared/app-config.ts, emits an idempotent SQL seed migration that UPSERTs
// each top-level key of APP_CONFIG into the app_config table as jsonb.

import { writeFileSync, mkdirSync } from 'node:fs'
import { resolve, dirname } from 'node:path'
import { fileURLToPath } from 'node:url'
import { createRequire } from 'node:module'
import { build } from 'esbuild'

const __dirname = dirname(fileURLToPath(import.meta.url))
const repoRoot = resolve(__dirname, '..')
const srcConfig = resolve(repoRoot, 'shared/app-config.ts')
const outMigration = resolve(repoRoot, 'supabase/migrations/20260509000001_app_config_seed.sql')

// Compile shared/app-config.ts to a temp JS module and import it.
const tmp = resolve(repoRoot, 'node_modules/.cache/app-config.mjs')
mkdirSync(dirname(tmp), { recursive: true })
await build({
  entryPoints: [srcConfig],
  outfile: tmp,
  format: 'esm',
  platform: 'node',
  bundle: true,
  logLevel: 'error',
})
const { APP_CONFIG } = await import(tmp + '?ts=' + Date.now())

// Emit one UPSERT per top-level key.
const stmts = Object.entries(APP_CONFIG)
  .map(([key, value]) => {
    const json = JSON.stringify(value).replace(/'/g, "''")
    return `INSERT INTO public.app_config(key, value) VALUES ('${key}', '${json}'::jsonb)
  ON CONFLICT (key) DO UPDATE SET value = EXCLUDED.value;`
  })
  .join('\n\n')

const header = `-- AUTO-GENERATED by scripts/gen-config.mjs from shared/app-config.ts
-- DO NOT EDIT BY HAND. Run \`pnpm gen:config\` to regenerate.

`

writeFileSync(outMigration, header + stmts + '\n')
console.log('Wrote', outMigration)
```

- [ ] **Step 2: Create `src/lib/otel.ts`**

```ts
import { trace, type Tracer } from '@opentelemetry/api'
import {
  WebTracerProvider,
  BatchSpanProcessor,
  NoopSpanProcessor,
} from '@opentelemetry/sdk-trace-web'
import { resourceFromAttributes } from '@opentelemetry/resources'
import { ATTR_SERVICE_NAME, ATTR_SERVICE_VERSION } from '@opentelemetry/semantic-conventions'
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http'
import { registerInstrumentations } from '@opentelemetry/instrumentation'
import { FetchInstrumentation } from '@opentelemetry/instrumentation-fetch'

let tracer: Tracer | null = null

/**
 * Initialise OpenTelemetry tracing.
 * If VITE_OTEL_EXPORTER_OTLP_ENDPOINT is unset, a NoopSpanProcessor is used so
 * spans are still created but dropped silently. Same code path in production
 * whether an OTEL backend is configured or not.
 */
export function initOtel(): Tracer {
  if (tracer) return tracer

  const endpoint = import.meta.env.VITE_OTEL_EXPORTER_OTLP_ENDPOINT
  const serviceName = import.meta.env.VITE_OTEL_SERVICE_NAME ?? 'sd-site-frontend'
  const serviceVersion = import.meta.env.VITE_OTEL_SERVICE_VERSION ?? '0.0.0'

  const resource = resourceFromAttributes({
    [ATTR_SERVICE_NAME]: serviceName,
    [ATTR_SERVICE_VERSION]: serviceVersion,
  })

  const spanProcessor = endpoint
    ? new BatchSpanProcessor(new OTLPTraceExporter({ url: endpoint }))
    : new NoopSpanProcessor()

  const provider = new WebTracerProvider({ resource, spanProcessors: [spanProcessor] })
  provider.register()

  registerInstrumentations({ instrumentations: [new FetchInstrumentation()] })

  tracer = trace.getTracer(serviceName, serviceVersion)
  return tracer
}

export function getTracer(): Tracer {
  return tracer ?? initOtel()
}
```

## Carry-over to Plan 02

Smells flagged during Plan 01 execution that didn't block completion but should be addressed early in Plan 02:

- **CI drift-check for generated files.** `pnpm gen:types` and `pnpm gen:config` aren't verified in CI. Schema or config changes could silently desync the committed generated files. Add `pnpm gen:types && git diff --exit-code shared/db-types.ts` and the equivalent for the seed migration.
- **`gen-config.mjs` SQL escaping.** The script builds SQL via plain string interpolation, only escaping `'`. If a config string ever contains a backslash or Unicode line separator, the generated SQL will break. Swap to dollar-quoted strings (`$cfg$...$cfg$`) before any user-facing config values land.
- **`initI18n()` is fire-and-forget.** Returns a discarded promise. Real consumers may want to await it; consider returning the promise.
- **React Router v7 idiomatic imports.** v7 prefers `react-router` over `react-router-dom`. Migrate as a small refactor.
- **Bundle size already 391KB minified** for a near-empty app (OTEL + i18next + router). Worth a code-splitting pass once real routes land.
- **Supabase CLI version pinning.** Local is 2.78.1, CI uses `latest`. Pin a version once schemas land in Plan 02 to avoid CLI behaviour drift.
