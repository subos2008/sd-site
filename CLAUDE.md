# Repo-wide guidance for Claude

Conventions, preferences, and gotchas that apply to anything Claude does in this repo. Read `README.md` and `docs/superpowers/plans/README.md` for the project overview and milestone status.

## Working preferences

- **Never excuse a failing check as "pre-existing".** If `pnpm lint`, `pnpm typecheck`, `pnpm test`, `pnpm test:db`, or `pnpm test:e2e` reports a failure or warning, fix it as part of the current task. Don't categorise the failure as "outside the task's scope" or "Plan N's problem". If a fix is genuinely out of reach (third-party bug, requires architectural change), name the blocker and ask before deferring.
- **Don't mention yourself in git commit messages.** No `Co-Authored-By: Claude…` lines, no "generated by" lines. Plain commit messages only.
- **Use only emojis the user explicitly requests.** No decorative emoji in commit messages, plan docs, code comments, or REAMDE entries.
- **Push after every commit.** GitHub remote is `git@github.com:subos2008/sd-site.git`. Frequent pushes are the cheapest backup. (The repo used to live under Dropbox cloud storage, which corrupted `.git/objects` mid-session before — see Plan 02 history; it was moved to `~/sd-site` outside any sync folder on 2026-07-12. Do NOT put a working git repo back inside Dropbox/iCloud/OneDrive.)

## Build / sandbox quirks on this machine

- **`PATH` shadowing with arm64 Node:** do NOT prepend `PATH=/usr/bin:/bin:/usr/local/bin:$PATH` to `pnpm build` on this machine. The Homebrew x64 Node shadows the arm64 nvm Node and breaks `@rollup/rollup-darwin-*` native bindings. For `pnpm test`, `pnpm test:db`, etc. either form is fine.
- **`supabase db reset` 502 hiccup:** the Supabase CLI 2.78.1 often reports `Error status 502: invalid response from upstream server` during the post-reset container restart. Migrations themselves apply cleanly; verify with `supabase status` and a `psql` query. Not a code bug.
- **Service-role-key guard:** `scripts/seed-dev-users.mjs` and `e2e/helpers/admin-signup.ts` refuse to run unless `SUPABASE_URL` resolves to `127.0.0.1`, `localhost`, or `::1`. The user maintains multiple Supabase projects locally and wants this hard safety net. Keep the guard in any new script that takes a service-role key.

## Plan/execution workflow

- The implementation milestones live in `docs/superpowers/plans/`. Each plan is independently executable.
- **Writing a new plan:** invoke `superpowers:writing-plans`. The plan body should contain real code in every step (no placeholders / TBDs), document open questions with sensible defaults, and include a `### Plan N execution deviations` section that the executor fills as the spec meets reality.
- **Executing a plan:** invoke `superpowers:subagent-driven-development`. Fresh subagent per task; two-stage review (spec compliance then code quality) after each. Log every deviation in the plan file as a separate commit, before moving on.
- **Test fixtures + RLS:** authenticated role cannot INSERT into `auth.users`. Any pgTAP fixture must `INSERT INTO auth.users` (and any RLS-protected reference data) BEFORE `SET LOCAL ROLE authenticated`. Similarly, switching `SET LOCAL ROLE anon` to test the unauthenticated path requires `SET LOCAL "request.jwt.claim.sub" = ''` first — `RESET ROLE` does not clear other GUCs.
- **Storage signed URLs:** the local Supabase image (CLI 2.78.1) does not expose `storage.create_signed_url` / `storage.create_signed_upload_url` as SQL helpers. All view RPCs return storage **paths**, not signed URLs; the frontend calls `supabase.storage.from('media').createSignedUrl(path, 3600)` itself. Storage RLS (`supabase/migrations/20260510000007_storage_rls.sql`) is what makes this work.

## Tone and output

- Default response length is short. Pre-commit checks and plan execution are exceptions — those merit detailed reporting.
- When in doubt about scope, ask before doing destructive or remote-touching things (`git push --force`, `supabase db reset` against anything other than the local sandbox, posting to GitHub Issues / PRs).
